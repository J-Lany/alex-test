# Криптоанализатор на Java

Небольшой учебный проект по работе с **шифром Цезаря**.

Проект задуман как **понятная и расширяемая болванка**: без лишней сложности, но с такой структурой, которая не развалится, когда появятся новые режимы, проверки и дополнительные классы.

## Что находится в этом README

- краткое описание проекта;
- полное описание задания в сворачиваемом блоке;
- структура папок проекта в сворачиваемом блоке;
- простое объяснение структуры проекта и того, зачем она нужна на практике.

## Идея проекта

Нужно написать программу, которая работает с **шифром Цезаря**.

Шифр Цезаря — это способ шифрования, при котором каждый символ текста сдвигается на фиксированное количество позиций в алфавите.

Пример:
- при сдвиге на `3` буква `А` превращается в `Г`;
- буква `Б` превращается в `Д`;
- и так далее.

## Криптографический алфавит

В проекте используется свой криптографический алфавит:

- все буквы русского алфавита;
- знаки пунктуации: `. , " : - ! ?`
- пробел.

Если во входном тексте встречаются символы, которых нет в этом алфавите, программа их пропускает.

## Обязательные режимы работы

### 1. Шифрование / расшифровка

Программа должна:
- получать путь к текстовому файлу;
- читать исходный текст;
- шифровать или расшифровывать его по заданному ключу;
- создавать новый файл с результатом.

### 2. Криптоанализ методом brute force

Программа должна:
- получать путь к зашифрованному тексту;
- пытаться подобрать ключ перебором;
- автоматически расшифровать текст.

Важно заранее продумать критерий, по которому программа поймёт, что ключ найден правильно.
Например:
- в тексте появились нормальные пробелы между словами;
- знаки препинания стоят в более реалистичных местах;
- текст стал похож на осмысленный русский текст.

## Дополнительные требования (опционально)

### 1. Статистический анализ

Дополнительно можно реализовать второй способ криптоанализа:
- пользователь загружает ещё один текст;
- программа собирает статистику по символам;
- затем использует эту статистику для попытки расшифровки зашифрованного текста.

### 2. User Interface

Можно сделать любой интерфейс:
- консольный;
- Swing;
- JavaFX.

Для первой версии проекта достаточно консольного интерфейса.

## Главная цель этой болванки

Сделать проект так, чтобы:
- было понятно, где точка входа;
- было легко добавить новые режимы;
- чтение/запись файлов не смешивались с логикой шифрования;
- brute force не был вперемешку с меню и вводом пользователя.

## Почему путь выглядит так: `src/main/java/com/owner/cryptanalyzer/`

На первый взгляд кажется, что это слишком много папок подряд, но на практике у каждой из них есть простая роль.

- `src` — здесь лежит исходный код проекта.
- `main` — здесь находится основной код программы, не тесты.
- `java` — здесь лежат именно Java-классы.

Связка `src/main/java` — это **стандартная и общепринятая структура Java-проекта**.  
Для нормальной практики её лучше воспринимать как обязательную, потому что к такой структуре привыкли разработчики, IDE и инструменты сборки.  
То есть технически Java не заставляет использовать именно её, но на практике это стандарт, которого лучше придерживаться с самого начала.

- `com/owner/cryptanalyzer` — это уже путь пакета.

То есть эта “цепочка папок” — не случайный набор, а стандартный способ показать:

1. где лежит код проекта;
2. какой код относится к основной программе;
3. как называется пакет у классов.

### Зачем это делается на практике

Так проект становится понятнее и аккуратнее:

- сразу видно, где основной код;
- легко отделять его от тестов и других файлов;
- классы не лежат все в одной куче;
- проще ориентироваться, когда проект растёт.

### Что такое `com/owner/cryptanalyzer`

Это фактически папки, которые соответствуют имени пакета.
Здесь `owner` — это не конкретно “твоё имя”, а часть, которая обозначает владельца проекта.
В зависимости от ситуации там может быть:
- компания — если это рабочий проект;
- username / ник / фамилия — если это личный или учебный проект;
- организация или общее пространство имён — если так принято в команде.

### Важно понимать

`owner` — это не обязательное слово.  
Это просто заглушка, которую нужно заменить на что-то своё.

Например, можно использовать:

- `com.tbank.cryptanalyzer`
- `com.shcherbakova.cryptanalyzer`

Также возможен более короткий вариант:
- `com.cryptanalyzer`
Но в таком случае промежуточной папки `owner` уже не будет.
- `com.cryptanalyzer`

Например, если в файле написано:

```java
package com.owner.cryptanalyzer;
```

то сам файл должен лежать в папке:

```text
src/main/java/com/owner/cryptanalyzer/
```

Если пакет будет глубже, например:

```java
package com.owner.cryptanalyzer.ui;
```

то файл уже должен лежать здесь:

```text
src/main/java/com/owner/cryptanalyzer/ui/
```

### Почему нельзя просто сложить все классы в одну папку

Можно, но только пока проект совсем маленький.

Когда классов становится больше, без структуры начинается путаница:
- где меню;
- где логика шифрования;
- где работа с файлами;
- где проверки;
- где точка входа.

Поэтому на практике код раскладывают по папкам и пакетам, чтобы каждая часть проекта лежала на своём месте.

### Что означают основные папки внутри проекта

#### `Application.java`

Это точка входа в программу.
Именно отсюда всё начинается: программа запускается, создаёт нужные объекты и передаёт управление дальше.

Проще говоря: это главный стартовый файл.

#### `ui/`

Здесь лежит всё, что связано с общением с пользователем.

Например:
- показать меню;
- спросить путь к файлу;
- спросить ключ;
- показать результат.

Важно: **UI не должен сам шифровать текст**.  
Его задача — принять команды от пользователя и передать их дальше.

#### `cipher/`

Здесь находится логика шифра Цезаря.

Именно этот пакет отвечает за вопросы:
- как сдвигать символы;
- как шифровать текст;
- как расшифровывать текст.

Важно: здесь не должно быть меню, чтения файла из консоли и прочей побочной логики.

#### `analysis/`

Здесь лежат способы взлома шифра.

Почему это вынесено отдельно:
- brute force — это не то же самое, что обычное шифрование;
- статистический анализ — это ещё один отдельный сценарий;
- если всё смешать в одном классе, код быстро станет тяжёлым и запутанным.

`BruteForceAnalyzer` — отвечает за перебор ключей.

`StatisticalAnalyzer` — за дополнительный режим, если ты решишь его делать позже.  
Даже если пока он будет пустой или не реализован до конца, место под него уже предусмотрено.

#### `io/`

Здесь работа с файлами.

Например:
- прочитать текст из файла;
- записать результат в новый файл.

Это важно вынести отдельно, потому что:
- шифрование — это одна задача;
- работа с файлами — другая задача.

Когда логика разделена, программу легче тестировать, чинить и расширять.

#### `alphabet/`

Здесь удобно хранить криптографический алфавит.

Например:
- строку или список всех допустимых символов;
- методы для поиска индекса символа;
- проверки, входит ли символ в алфавит.

Это лучше, чем разбрасывать алфавит по разным классам в виде случайных строк.

#### `model/`

Здесь находятся простые вспомогательные сущности проекта.

Например:
- `OperationMode` — перечисление режимов (`ENCRYPT`, `DECRYPT`, `BRUTE_FORCE`, `STATISTICAL_ANALYSIS`);
- `AnalysisResult` — объект, в котором можно хранить результат анализа: найденный ключ, расшифрованный текст, признак успеха.

Это нужно, чтобы не таскать по всему проекту хаотичный набор строк, чисел и флагов.

#### `validation/`

Здесь проверки входных данных.

Например:
- путь к файлу не пустой;
- файл существует;
- ключ находится в допустимом диапазоне;
- текст не пустой.

Почему это полезно:
если проверки разбросаны по всему проекту, их сложно поддерживать.
Когда они собраны в одном месте, логика чище.

#### `util/`

Здесь лежат небольшие вспомогательные вещи, которые не относятся напрямую к одному конкретному блоку.

Например:
- нормализация текста;
- мелкие операции со строками;
- общие служебные методы.

Важно: сюда не нужно складывать всё подряд.  
`util` — это не мусорка, а пакет для действительно небольших общих помощников.

#### `resources/`

Папка под дополнительные ресурсы.

В первой версии проекта она может быть пустой.
Позже туда можно положить:
- тестовые тексты;
- шаблоны;
- вспомогательные данные.

#### `src/test/java`

Здесь лежат тесты.

Даже если ты пока не пишешь тесты активно, саму папку лучше завести сразу.  
Тогда структура проекта с самого начала будет взрослой и привычной.

### Почему эта структура хороша именно для учебного проекта

Она хороша потому, что:

1. **Не слишком сложная.**  
   Здесь нет лишней многослойности и “энтерпрайз-магии”.

2. **Не всё свалено в один класс.**  
   У каждой части есть своя зона ответственности.

3. **Проект можно расширять.**  
   Сегодня есть только шифрование и brute force, а завтра можно добавить статистику, другой интерфейс или новые проверки.

4. **Легче понимать код.**  
   Когда ты открываешь проект, ты сразу видишь: где меню, где файлы, где шифр, где анализ.

5. **Это хорошая подготовка к нормальной Java-структуре.**  
   Такой подход уже ближе к реальным проектам, чем один огромный класс на 700 строк.

Самое главное правило такое:

**путь папок должен совпадать с тем, что написано в `package` в начале файла.**

### Что важно не делать

- не смешивать ввод пользователя, работу с файлами и шифрование в одном классе;
- не складывать весь код в `Main`;
- не называть свои классы `FileReader` и `FileWriter`, потому что такие классы уже есть в Java;
- не делать слишком много пустых абстракций “на будущее”, если они пока реально не нужны.

## Замечание по репозиторию

Для старта достаточно обычного публичного GitHub-репозитория с:
- `README.md`;
- `.gitignore`;
- папкой `src`.

Позже, если проект будет переводиться на Maven или Gradle, можно добавить:
- `pom.xml` — для Maven;
- или `build.gradle` / `settings.gradle` — для Gradle.

То есть GitHub-репозиторий не требует какой-то “специальной java-настройки”.
Главное — нормальная структура проекта и понятные файлы внутри.

## Ближайший следующий шаг

После создания болванки можно переходить к следующему уровню:

1. определить точку входа (`Application`);
2. продумать сценарий меню;
3. описать, какие методы должны быть у `CaesarCipher`;
4. отдельно продумать критерий успешного brute force;
5. только потом писать реализацию.
